## Bugs Fixed

### Bug 1: Sparkline showing 120% adherence
**Cause:** 10 protocols counted as 10 doses (missing `jsonb_array_length(time_schedule)`)  
**Fix:** Use `jsonb_array_length(time_schedule)` to count actual dose slots (2 for "08:00", "20:00")  
**Result:** 10 protocols, 12 doses/day → 100% adherence ✅

### Bug 2: Heatmap showing 900% adherence  
**Cause:** Accumulated all 90 days of logs vs expected_doses for single occurrence  
**Fix:** Added `day_occurrences` CTE to scale expected_doses by weeks occurred  
**Result:** Correct per-period adherence (not 900%) ✅

### Bug 3: Heatmap never rendered (hasEnoughData: false)
**Cause:** Threshold ≥20 cells impossible for users with only morning+night (max 14 cells)  
**Fix:** Reduced threshold from 20 → 7 filled cells  
**Result:** Heatmap renders for users with morning+night schedule ✅

### Bug 4: Sparkline capped at 30 days despite 90-day load
**Cause:** Hard-coded `daysCount = size === 'expanded' ? 30 : 7` in component  
**Fix:** Added `days` prop to SparklineAdesao, retained 30-day default for UI  
**Result:** Sparkline correctly processes 30 days from 90-day data ✅

## Architecture

**Pattern:** Server-side pre-aggregation (database views) + light client-side consumption  
**Benefit:** O(N) → O(1) lookups, zero processing on mobile  
**Trade-off:** Database complexity increased (+100 lines SQL), client complexity decreased (-50 lines JS)

## Learnings (R-NNN, AP-NNN)

### R-121: Zod Validation for Service Parameters
**Rule:** All service functions accepting parameters MUST use Zod `safeParse()` + schema  
**Why:** Prevents silent failures; documents input contract; catches edge cases  
**Applied:** `getDailyAdherenceFromView(days)` — schema validates `0 < days ≤ 365`

### R-122: Extract Functions at 30-Line Threshold
**Rule:** If a function exceeds 30 lines, extract helpers to reduce cognitive load  
**Why:** Improves readability, testability, reusability  
**Applied:** `buildHeatmapNarrative()` + `buildAdherenceGrid()` extracted from `getAdherencePatternFromView()`

### AP-121: Don't Count DISTINCT Protocols When You Need Doses
**Anti-Pattern:** Using `COUNT(DISTINCT protocol_id)` when dose count is needed  
**Symptom:** 10 protocolos → "10 doses" even if each has 2 doses scheduled  
**Fix:** Use `jsonb_array_length(time_schedule)` to count actual dose slots  
**Root Cause:** Mixing domain concepts (protocols vs doses)

### AP-122: Watch Out for Cartesian Products in Aggregation
**Anti-Pattern:** LEFT JOIN logs ⨝ expected_doses → SUM() without proper grouping  
**Symptom:** 12 logs × 10 protocols = 120 rows → SUM() sums duplicates  
**Fix:** Pre-aggregate expected_doses BEFORE joining with logs  
**Root Cause:** Missing CTE grouping step
