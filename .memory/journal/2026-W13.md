## 📚 Rules & Learnings

### New Rules Added

- **R-137:** Typography weights must be ≥400 for accessibility (no light/thin fonts for elderly patients)
- **R-138:** lucide-react icons always paired with text labels (accessibility + UX clarity)
- **R-139:** CSS `@import` for fonts in token files, never `<link>` in index.html (scoped delivery)
- **R-140:** Heading specific weights via `--heading-N-weight` variables (level-appropriate hierarchy)

### Anti-Patterns Reinforced

- **AP-D01:** Using font weights <400 → breaks WCAG 2.1 AA for geriatric users
- **AP-D02:** Adding fonts via `<link>` in index.html → affects all users (breaks gradual rollout)
- **AP-D03:** Icon-only UI without text label → accessibility failure + UX confusion

---

## 🎓 Lessons Learned

### Challenge 1: Backward Compatibility vs. New Design
- **Problem:** Spec required size aliases (`--text-xs` through `--text-5xl`) for backward compat
- **Learning:** Token systems must layer (semantic names + numeric aliases) to avoid breaking existing components
- **Applied:** All tokens added both ways (display/headline/title AND text-4xl/3xl/2xl)

### Challenge 2: Form Element Inheritance
- **Problem:** Weren't sure if inputs/buttons would inherit font from container
- **Learning:** CSS `font-family` inherits automatically; explicit selector for form elements ensures clarity
- **Applied:** Added explicit `[data-redesign="true"] input, textarea, select, button` rules

### Challenge 3: Typography Scale Completeness
- **Problem:** Missing heading defaults that could cause h1-h6 visual inconsistency
- **Learning:** Define ALL heading variables upfront (size + weight + line-height + tracking per level)
- **Applied:** 6 heading levels × 2-3 properties each = comprehensive hierarchy

## 🎨 Architecture Highlights

### Material 3 "No-Line Rule"
- Hierarchy via tonal backgrounds (Level 0 #f8fafb → Level 1 #f2f4f5 → Level 2 #ffffff)
- No `border: 1px solid` — separation through color difference instead
- Classes enforcing: `.surface`, `.surface-container-low`, `.surface-container-lowest`

### Scoping Strategy
- **New classes (safe, aditivo):** `.page-container`, `.grid-*`, `.app-main` — no scoping
- **Existing names (collision risk):** `.page-title`, `.section-header`, `.icon-container` — scoped with `[data-redesign="true"]`
- **Result:** Zero breaking changes; gradual rollout ready

### Mobile-First Responsive
- Mobile (<768px): 1 column, 1rem padding, BottomNav (80px clearance)
- Tablet (768-1023px): 2 columns, 1.5rem padding, sidebar appears
- Desktop (≥1024px): 2-3 columns, 2rem padding, 16rem sidebar, max-width 80rem

## 📚 New Rules Registered

- **R-138:** Class name collision prevention — scan existing codebase before adding utility classes
- **R-139:** CSS selector grouping — group selectors sharing 3+ common properties
- **R-140:** Multiline CSS formatting — 3+ properties → separate lines

## 🚫 Anti-Patterns Prevented

- **AP-W02:** Over-scoping feature flags — only scope when there's actual naming collision, not all new classes

## 🎓 Key Patterns & Decisions

### R-150: Dashboard Branching with Feature Flag
- **Context:** Need to ship DashboardRedesign without breaking existing Dashboard
- **Pattern:** `isRedesignEnabled ? <DashboardRedesign /> : <Dashboard />`
- **Benefit:** A/B testing capable, zero risk rollback, users can compare UIs
- **Cost:** Duplicate UI logic (acceptable, views are isolated)

### R-151: Lazy Component Imports in Conditional Branches
- **Context:** DashboardRedesign is only loaded when flag is enabled
- **Pattern:** `const DashboardRedesign = lazy(...)` at module level + `<Suspense>` at render time
- **Benefit:** Main bundle unaffected (redesign chunks load only if needed)
- **Gotcha:** Vite validates lazy import paths at build time; file must exist before App.jsx is built

### R-152: Shared Hooks Pattern for Dashboard Variants
- **Context:** Both Dashboard and DashboardRedesign need same data (stats, zones, stock)
