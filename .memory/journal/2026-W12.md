### UX: Redução de ruído visual no StockCard (PR #402)

**Problema:** StockCard exibia todas as entradas de estoque com o mesmo peso visual:
1. Lotes com `quantity = 0` (zerados por FIFO via `stockService.decrease()`)
2. Entradas geradas automaticamente por `logService.delete()` → `stockService.increase(id, qty, "Dose excluída (ID: X)")`

**Fix:** Classificação das entradas em 3 tipos antes de renderizar:
- **Lotes ativos**: `quantity > 0` e `notes` não é ajuste de sistema → exibidos normalmente
- **Lotes consumidos**: `quantity === 0` → ocultos por padrão (seção colapsável)
- **Ajustes automáticos**: `notes` começa com `"Dose excluída"` ou `"Ajuste de dose"` → ocultos por padrão

**Padrão aplicado:**
```javascript
const SYSTEM_NOTE_PREFIXES = ['Dose excluída', 'Ajuste de dose']
const isSystemAdjustment = (e) => SYSTEM_NOTE_PREFIXES.some(p => e.notes?.startsWith(p))
const { activeEntries, consumedEntries, systemEntries } = stockEntries.reduce(...)
```

# Sprint 8.3.1 — Bugfix Chatbot Hallucinations

**Data:** 2026-03-20 (pós-deploy)
**PR:** #408 — mergeado em main
**Status:** ENTREGUE ✅
**Commit:** `1e47cfb`

---

## Problema

Chatbot alucinava respostas farmacológicas — quando perguntado "qual a função do Selozok?", o LLM respondeu "Sertralina (antidepressivo)" em vez de "Metoprolol Succinato (betabloqueador)". Erro perigoso num contexto médico.

## Análise das Causas

1. **Falta de grounding farmacológico:** contextBuilder enviava só `SeloZok (50mg)` sem `active_ingredient` ou `therapeutic_class`. LLM tinha que recorrer ao conhecimento de treinamento — onde errou.
2. **Parâmetros de inferência muito altos:** `temperature: 0.7, top_p: 0.9` faziam o modelo amostrar de distribuição ampla, aumentando chance de "inventar" quando contexto incompleto.
3. **Modelo base:** `llama-3.3-70b-versatile` tem reputação moderada em factual recall (não é o melhor para farmacologia).

## Solução (Opções E + Ajustes Parametros + D)

### 1. Grounding com Contexto Farmacológico (Opção E)
- Adicionar `active_ingredient` e `therapeutic_class` ao contexto do paciente
- **Antes:** `- SeloZok (50mg): diario, ...`
- **Depois:** `- SeloZok [Succinato de Metoprolol, Betabloqueador] (50mg): diario, ...`
- Garantir `null` ou string vazia não expõe texto "null" no contexto (usar `filter(Boolean)`)

### 2. Ajuste de Parâmetros (Temperature + Top_p)
- `temperature: 0.7 → 0.2` — respostas factuais, menos aleatoriedade
- `top_p: 0.9 → 1.0` — desativar nucleus sampling quando temperatura baixa
- Resultado: modelo menos propenso a inventar

### 3. Trocar Modelo (Opção D)
- Trocar para `groq/compound` (seleção inteligente do modelo conforme "pedido")
- Reduz dependência de modelo específico com baixo factual recall

## Arquivos Modificados

| Arquivo | Mudança |
|---------|---------|
| `api/chatbot.js` | `temperature: 0.2, top_p: 1.0` (linhas 57-58) |
| `src/features/chatbot/services/contextBuilder.js` | Incluir `principioAtivo` + `classeTerapeutica`; filtrar Boolean para omitir null |
| `src/features/chatbot/__tests__/contextBuilder.test.js` | Adicionar testes para `therapeutic_class` + ausência de null |
| `src/features/chatbot/__tests__/chatbotService.test.js` | Corrigir mock localStorage com `vi.stubGlobal()` (AP-T03) |

## Quality Gates

- ✅ 33/33 testes passando
- ✅ Nenhum `null` ou `undefined` exposto no contexto
- ✅ localStorage mock corrigido (AP-T03)
- ✅ Sem impacto em UX — mudanças internas apenas

## Lições Aprendidas

- **Alucinação = falta de grounding, não só prompt:** restricoes no prompt sozinhas não fazem o LLM deixar de inventar farmacologia. Precisa de contexto factual + parametros conservadores.
- **Dados do BD já existem:** `active_ingredient` + `therapeutic_class` já estavam nas medicines — só não estavam sendo enviados ao LLM.
- **Temperature baixa é crítico para medical chatbots:** `0.2` vs `0.7` muda drasticamente a disposição do modelo em "arriscar" respostas.
- **Modelo importa:** mesmo com tudo acima, qualidade do modelo base impacta (por isso opção D — groq/compound).

Otimizar custo de API e latência do Groq em TODOS os canais (web/PWA + Telegram) via Prompt Caching. Groq oferece 50% desconto em tokens que já foram processados (cached_prompt_tokens), permitindo reutilização de conteúdo estático em conversas multi-turn.

  - `server/bot/services/chatbotServerService.js` (Telegram)

**Resultado:**

- Contexto do paciente (dinâmico) vem DEPOIS → diferencia por conversa

#### 2. Logging de Cache Hit Rate

**Adicionado em:**
- `api/chatbot.js` (Vercel serverless para web)
- `server/bot/services/chatbotServerService.js` (server Telegram)

**Métricas capturadas:**
- `cachedTokens` = tokens reutilizados de cache (Groq retorna via `cached_prompt_tokens`)
- `cacheHitRate` = (cachedTokens / promptTokens) * 100%
- `estimatedTokenSavings` = cachedTokens * 0.5 (desconto simulado)

**Exemplo de log (Vercel):**
```json

  "message": "Groq response received",
  "promptTokens": 150,
  "cachedTokens": 75,
  "cacheHitRate": "50%",
  "estimatedTokenSavings": 37
}
```

**Exemplo de log (Telegram/Node.js):**
```

estimatedTokenSavings: 37
```

## Sprint 8.5 — Debugging & Bug Fix: Chatbot IA no Telegram

**Data:** 2026-03-20 (pós-deploy Sprint 8.3.2)
**PR:** #412 | **Commits:**
- `f30a72f` — Added structured logging Vercel-friendly in api/telegram.js
- `db69312` — Added routing fallback in conversational.js to call handleChatbotMessage
- `12470bd` — Added sendChatAction method to bot adapter
**Status:** ✅ ENTREGUE E MERGEADO

### Problema

User testou chatbot IA no Telegram e recebeu **"não aconteceu nada"** — sem resposta, sem erro, sem logs em Vercel. Impossível debugar remotamente.

### Root Cause Analysis (3 Camadas)

#### Layer 1: Logs Invisíveis em Vercel (Observabilidade)

**Problema:** Logging estruturado existente em `server/bot/commands/chatbot.js` e `server/bot/services/chatbotServerService.js` não aparecia nos logs do Vercel.

**Raiz:** Logs em Node.js server context (event loop do bot) vs logs em Vercel serverless context (API handler). Dois contextos diferentes.

**Insight:** `api/telegram.js` é o **entry point Vercel** onde os logs REALMENTE são capturados. Logs em `server/bot/**` rodavam em contexto Node.js puro, invisível para Vercel logging.

**Fix:** Adicionar `import { createLogger } from '../server/bot/logger.js'` no topo de `api/telegram.js` e logar em 3 pontos estratégicos:
- Entrada webhook: tipo de update (message vs callback_query)
- Roteamento: qual comando foi detectado (ou fallback conversacional)
- Resultado: sucesso ou erro

**Regra:** Logging estruturado em serverless functions DEVE estar no `.js` do `api/` directory, não em níveis inferiores.

#### Layer 2: Message Routing Incompleto (Lógica)

**Problema:** `bot.on('message', ...)` listener em `server/bot/callbacks/conversational.js` só processava mensagens com `waitingForInput: true` (sessão ativa). Mensagens de conversa livre (não-comando, sem sessão) eram **silenciosamente ignoradas**.

**Raiz:** Spec de Sprint 8.3.2 documentou "reutilizar handleChatbotMessage para Telegram" mas não atualizou o listener para chamar fallback.

**Fix:** Adicionar `else` clause que chama `handleChatbotMessage(bot, msg)` quando:
```javascript
} else if (!sessionData?.waitingForInput) {
  // Fallback: mensagem sem session ativa → processar como chatbot IA
  logger.info('🤖 Mensagem sem session ativa → chatbot IA', { chatId })
  await handleChatbotMessage(bot, msg)
}
```

**Regra:** Event-driven bot architectures com múltiplos handlers PRECISAM de fallback explícito para casos não-capturados por handlers específicos.

#### Layer 3: Bot Adapter Incompleto (API Compatibility)

**Problema:** Erro no Vercel: `bot.sendChatAction is not a function`. ChatWindow usava `bot.sendChatAction(chatId, 'typing')` para mostrar indicador de digitação, mas o adapter mock em `api/telegram.js` não implementava esse método.

**Raiz:** Mock bot adapter em `api/telegram.js` tinha: sendMessage, editMessageText, answerCallbackQuery, deleteMessage, getMe, on, _emit. Faltava: sendChatAction.

**Fix:** Adicionar método ao adapter:
```javascript
sendChatAction: async (chatId, action) => {
  return telegramFetch('sendChatAction', { chat_id: chatId, action })
}
```

**Regra:** Mock/adapter objects precisam ter interface **COMPLETA** de todos os métodos que os handlers vão chamar. Testar localmente se a interface está sendo usada.

### Solução Aplicada

| Camada | Arquivo | Mudança | Impacto |
|--------|---------|---------|--------|
| 1 | `api/telegram.js` | +38 linhas logging estruturado em 3 pontos | Logs agora visíveis em Vercel |
| 2 | `server/bot/callbacks/conversational.js` | +7 linhas fallback + import handler | Mensagens livres routeadas a chatbot |
| 3 | `api/telegram.js` | +4 linhas sendChatAction method | Typing indicator funcionando |

Sequência de log no Vercel prova fluxo completo:

### Lições Aprendidas

1. **Observabilidade é crítica em serverless:** Sem logs visíveis, debugging remoto é impossível. Estruturar logs no entry point Vercel, não em níveis inferiores.

2. **Logging em diferentes contextos:** Node.js server context vs Vercel serverless context são invisíveis um para o outro. Sempre logar onde o log será capturado (para Vercel: `api/` directory).

3. **Event-driven routers precisam de fallback explícito:** Quando múltiplos listeners competem, sempre ter um fallback `else` catch-all para mensagens não-capturadas.

4. **Mock/adapter interface completeness:** Testar localmente que TODOS os métodos chamados pelos handlers existem no mock. Não assumi interface completa.

5. **Multi-channel chatbot:** Web (DashboardContext + localStorage rate limit) vs Telegram (Supabase queries + Map rate limit) precisam de adapters separados. Lógica de negócio (contextBuilder, safetyGuard, Groq) pode ser reutilizada.
