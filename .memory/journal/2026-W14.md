## In Progress / Delivered — Stock Purchases Refactor (redesign-first)

### Escopo entregue
- Migration de estoque/purchases aplicada com sucesso no Supabase: separação de `purchases`, `stock_adjustments` e `stock_consumptions`, além de `medicines.regulatory_category`
- Refactor dos serviços de estoque e logs para usar RPCs transacionais (`create_purchase_with_stock`, `consume_stock_fifo`, `restore_stock_for_log`, `apply_manual_stock_adjustment`)
- Novo `purchaseService` como fonte canônica de histórico de compras, última compra e preço médio
- Redesign `?redesign=1` migrado para ler histórico/última compra a partir de `purchases` em vez de inferir compras por `stock.notes`

- Fluxos ANVISA propagando `regulatory_category` no ETL e nos formulários de medicamento/tratamento
- Telegram bot migrado para os novos RPCs para compra e consumo FIFO, removendo mutação manual direta de `stock`

### Decisões tomadas
1. O rollout funcional desta onda fica limitado à superfície redesign (`?redesign=1`), mesmo com services compartilhados
2. Histórico de compra e última compra passam a vir exclusivamente de `purchases`; `stock` permanece fonte de saldo corrente
3. O bot Telegram entra na mesma onda de arquitetura para evitar divergência entre web e automações
4. `regulatory_category` de ANVISA é obrigatório nesta entrega para suportar a UX de laboratório por compra
5. Backfill de `stock_adjustments` não deve inserir linhas com `quantity_delta = 0`; a migration foi ajustada para filtrar `s.quantity <> 0`

### Validação
- `npm run validate:agent` ✅ (`32` arquivos de teste, `543` testes passando)
- Migration executada com sucesso no Supabase após ajuste no backfill de ajustes legacy

### Lições aprendidas
- Ao migrar modelo legado de estoque para auditoria separada, linhas históricas zeradas podem violar constraints novas se o backfill não filtrar explicitamente `0`
- O bot tinha acoplamento silencioso com FIFO client-side; alinhar Telegram aos mesmos RPCs do app evita drift arquitetural e bugs de restauração
