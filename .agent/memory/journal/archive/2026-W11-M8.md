## Technical Details

### Bug Pattern
When `isLoadingPatterns` changed (false → true → false) during async request:
1. React detected dependency change
2. Called `old(null)` (returned early without disconnect)
3. Called `new(element)` (created new observer)
4. 16ms window with BOTH observers alive
5. Both could fire `getAdherencePatternFromView()` simultaneously

### Solution Pattern
Ref callbacks should **always have empty deps `[]`** because:
- Return value is ignored by React (no cleanup)
- Stateful logic should use `useRef` instead of closing over state
- This prevents unnecessary recreation and race conditions

### Performance Impact
- Before: Variable number of simultaneous requests (1-2+)
- After: Exactly 1 request, guaranteed
- No perceptible UX change (heatmap loads when scrolling to it)

## Learnings

### R-120 Application (Ref Callback Pattern)
**Rule:** Ref callbacks always have `deps []`. Use `useRef` for stateful flags.

**Why:** Ref callbacks are called by React at mount/unmount, not during renders. The return value is ignored (cleanup only runs in useEffect). Dependencies change → callback recreated → race conditions.

### Confirmation of AP-P11 (Existing Anti-Pattern)
AP-P11 was already documented but this sprint provided real-world example:
- Pattern: `useCallback` with state in deps of ref callback
- Consequence: Duplicate event fires, race conditions
- Prevention: `deps []` + `useRef` for closure-stable flags
- Source: Sprint M8 — HealthHistory sentinel observer race condition
