# AP-115 — Incomplete Dispatch Schema Enum

## Problem

Em `dispatchNotification()` schema Zod, o enum `kind` não incluía os novos tipos de notificação groupada:
- `dose_reminder_by_plan` (bloco by treatment_plan)
- `dose_reminder_misc` (doses sem protocolo)
- `dose_reminder_individual` (dose isolada)

Resultado: Wave N1.1–N1.2 disparavam estes tipos de notificação mas o schema rejeitava silenciosamente (`safeParse().success = false`), falhando sem erro visível no log.

## Evidence

- Sprint N1.3: Debug revelou que todos os blocos agrupados iam para dedup queue mas nunca eram despachados
- Root cause: schema enum faltava os 3 novos `kind` values
- Impacto: Notificações grouped não chegavam ao usuário (silent failure)

## How It Happens

1. Backend cria blocos semânticos via `partitionDoses()`
2. Tenta dispatchar com `kind = 'dose_reminder_by_plan'`
3. Zod schema rejeita valor não-listado no enum
4. Nenhum erro é thrown (só `result.success = false`)
5. Notificação é silenciosamente descartada

## Prevention

**SEMPRE sincronizar schema enum com tipos de evento reais:**

```js
// Verificar: se há novo tipo de evento no backend, adicionar ao schema
const dispatchInputSchema = z.object({
  kind: z.enum([
    'dose_reminder',           // legacy
    'dose_reminder_by_plan',   // novo em N1
    'dose_reminder_misc',      // novo em N1
    'dose_reminder_individual' // novo em N1
    // ... outros tipos
  ])
})
```

Depois de adicionar novo `kind`:
1. Grep backend para `kind = '...'`
2. Verificar todos os matches estão no enum
3. Testar com `safeParse()` em unit test
