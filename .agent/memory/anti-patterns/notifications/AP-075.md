# AP-075 — Presentation Leakage in Business Logic (L1)

## O que é?
Construir mensagens de notificação (títulos, corpos, strings com emojis) dentro de arquivos que deveriam apenas calcular dados (ex: `server/bot/tasks.js`, `api/notify.js`).

## Por que é um problema?
1.  **Inflexibilidade**: Se o Telegram exige MarkdownV2 mas o Push iOS exige texto puro, a lógica de negócio fica "suja" com múltiplos formatos.
2.  **Duplicação**: Múltiplos disparadores acabam duplicando a lógica de "como formatar o nome do medicamento com a dose".
3.  **Dificuldade de Teste**: Testar se um lembrete foi gerado exige comparar strings complexas em vez de dados estruturados.

## Como evitar?
A camada L1 deve passar apenas objetos de dados crus para o `NotificationDispatcher`. A camada L2 (`buildNotificationPayload`) é a única responsável por decidir o "tom de voz" e o formato visual.

**ERRADO (L1 construindo strings):**
```javascript
// server/bot/tasks.js
const message = `💊 Hora de tomar seu ${med.name}!`; // NÃO: Emoji e formatação em L1
await dispatch({ kind: 'reminder', body: message });
```

**CERTO (L1 passando dados crus):**
```javascript
// server/bot/tasks.js
await dispatch({ 
  kind: 'dose_reminder', 
  data: { medicineName: med.name, scheduledTime: med.time } 
});
```
