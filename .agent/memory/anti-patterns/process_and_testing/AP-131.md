# [AP-131] Stale Mock Method Signature after Service Refactor

## Contexto
Durante a refatoração do `expoPushChannel.js`, o método interno do cliente Expo foi alterado ou o mock no teste estava usando um nome genérico (`.send()`) enquanto a implementação real chamava `.sendPushNotificationsAsync()`.

## Sintoma
O teste falha com erro de "method is not a function" ou simplesmente não captura a chamada (fail silently se o mock não for obrigatório), mas o erro real foi:
`expoClient.send is not a function`

## Causa Raiz
Ao refatorar serviços que dependem de bibliotecas externas (como `expo-server-sdk`), os testes unitários costumam mockar essas dependências. Se a implementação muda o método chamado mas o teste não é atualizado, ocorre uma regressão de teste.

## Como Evitar
1. **Interface de Mock Idêntica:** Garanta que o mock espelhe exatamente a API que está sendo utilizada no código.
2. **Spy over Mock:** Sempre que possível, use `vi.spyOn` em instâncias reais ou mocks tipados para garantir que o método existe.
3. **Busca por Consumidores:** Ao alterar uma chamada de método em um serviço, use `grep` para encontrar todos os arquivos de teste (`.test.js`) que mockam esse serviço ou sua dependência.
