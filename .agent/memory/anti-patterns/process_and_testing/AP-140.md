# AP-140: Strict Metadata Validation causing Logging Regressions

**Context:**  
During the notification architecture refactor (Gate 5), internal metadata schemas used for logging were enforced with `z.strict()`. While this is good for domain contracts, it caused a regression in system-wide traceability.

**Symptom:**  
Domain-specific fields like `medicineName`, `protocolId`, or `planId` were being stripped by the `metadataSchema` validation before reaching the `logService`, resulting in incomplete logs and broken DLQ traceability.

**Anti-Pattern:**  
Enforcing strict schemas on internal "passthrough" objects (like metadata) that serve as containers for cross-cutting concerns (logging, metrics, tracing).

**The Fix:**  
Use `.passthrough()` on Zod schemas for metadata and internal log payloads. This allows domain-specific information to propagate through the system without violating the base structural contract.

**Example (Zod):**
```javascript
// BEFORE (Regression)
const metadataSchema = z.object({
  channel: z.string(),
  // ...
}).strict();

// AFTER (Hardened for Traceability)
const metadataSchema = z.object({
  channel: z.string(),
  // ...
}).passthrough(); // Allows medicineName, protocolId, etc. to flow to logs
```
