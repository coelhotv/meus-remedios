# AP-168 — Cache invalidation incompleto em mutation (esquece snapshots adjacentes)

**Sintoma**: `useProtocolMutation.toggleActive` invalidava apenas `@dosiq/protocols-snapshot` (detail). Listagem `useTreatments` usa `@dosiq/treatments-snapshot` — sem invalidar, lista serviria cache stale com item ainda na tab antiga (Ativos↔Pausados) caso refresh on focus falhe (rede instável, race).

**Root cause**: implementação de mutation foca no cache do detalhe (mais óbvio) e esquece dos consumers laterais (listagens com cache key próprio). Padrão recorrente em mobile com AsyncStorage snapshots por consumer.

**Por que perigoso**: bug latente — 90% do tempo `useFocusEffect(refresh)` na listagem mascara o problema (fetch fresco do server independente do cache). Aparece só em rede instável + race condition específica.

**Fix aplicado**:

```js
// useProtocolMutation.toggleActive
await Promise.all([
  AsyncStorage.removeItem(PROTOCOLS_CACHE_KEY),    // detail
  AsyncStorage.removeItem(TREATMENTS_CACHE_KEY),   // listagem
]).catch(() => {})
```

**Regra preventiva** (ver [[R-236]]): toda mutation deve documentar inline (comment/JSDoc) TODOS os caches que invalida. Pattern:

```js
// Mutation X invalida snapshots:
//   - @dosiq/foo-snapshot (detail/useFoo)
//   - @dosiq/bar-snapshot (listagem/useBars)
//   - @dosiq/baz-snapshot (se efeito cascata em outro domínio)
```

**Outros lugares afetados** (verificar pós-Fase 2.5):
- `useProtocolMutation.create` — invalida só PROTOCOLS_CACHE_KEY (listagem useTreatments pode ficar stale após criar)
- `useProtocolMutation.update` — idem
- `useProtocolDelete` — idem
- `useMedicineMutation.*` (Fase 1) — idem

Sweep retroativo pode ser aplicado em fix-pack separado.
