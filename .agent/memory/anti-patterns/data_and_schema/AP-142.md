# AP-142 — FIFO com filtro entry_type restritivo cria "phantom stock"

## Problem

`consume_stock_fifo` foi implementado com `AND entry_type = 'purchase'` no loop e no pre-check de disponibilidade. Entradas `entry_type = 'adjustment'` com `quantity > 0` são incluídas em `medicine_stock_summary` (a view não filtra entry_type) mas são **invisíveis para o FIFO**.

Resultado: usuário vê saldo X na UI, mas qualquer tentativa de registrar dose lança `Estoque insuficiente` porque o saldo consumível real é X - phantom.

## Evidence

Auditoria de 2026-05-13 revelou 4 medicamentos afetados:

| Medicamento | FIFO consumível | UI mostrava | Status |
|---|---|---|---|
| SeloZok | 0 | 4 | **BLOQUEADO** |
| Clopidogrel | 58 | 60 | parcial |
| Olmesartana | 55 | 58 | parcial |
| Ácido Acetilsalicílico | 10 | 13 | parcial |

SeloZok: usuário possuía 4 comprimidos físicos reais (devolvidos ao estoque por exclusões de dose em mar/2026), mas não conseguia registrar nenhuma dose.

## Root Cause

O backfill da migração `20260402_stock_purchases_refactor.sql` converteu corretamente linhas antigas do tipo "Dose excluída / Ajuste de dose" para `entry_type = 'adjustment'`. Essas entradas representam pílulas REAIS devolvidas ao estoque. Porém o implementador da RPC adicionou um filtro `AND entry_type = 'purchase'` não previsto na spec (seção 7.2), que exclui essas entradas do consumo FIFO.

**A spec (seção 7.2) não inclui filtro de entry_type no pseudocódigo:**
```sql
-- spec (correto):
WHERE user_id = auth.uid()
  AND medicine_id = p_medicine_id
  AND quantity > 0

-- implementado (errado):
WHERE user_id = auth.uid()
  AND medicine_id = p_medicine_id
  AND quantity > 0
  AND entry_type = 'purchase'  -- ← adicionado pelo agente, não está na spec
```

## How It Happens

1. Migration backfill cria `adjustment` entries com `quantity > 0` (pílulas reais)
2. `apply_manual_stock_adjustment` também cria `entry_type = 'adjustment'` com `quantity > 0`
3. `medicine_stock_summary` view soma todos os `quantity > 0` independente de entry_type
4. UI exibe saldo inflado
5. FIFO ignora adjustment entries → "Estoque insuficiente"

## Prevention

**Ao implementar qualquer RPC de consumo/FIFO, o filtro de consumo DEVE ser:**

```sql
AND entry_type != 'legacy_unrecoverable'
-- NUNCA: AND entry_type = 'purchase'
```

`legacy_unrecoverable` = entradas sem dados para reconstrução histórica, sempre com `quantity <= 0`.
`adjustment` = ajustes positivos legítimos (manual ou backfill), DEVEM ser consumíveis.

**Regra geral:** se `medicine_stock_summary` inclui a entrada, o FIFO deve poder consumi-la.

## Fix Applied

Migration `fix_consume_stock_fifo_adjustment_entries` (2026-05-13):
- Ambos os overloads de `consume_stock_fifo` (client + server-side)
- Trocaram `AND entry_type = 'purchase'` por `AND entry_type != 'legacy_unrecoverable'`
- SeloZok phantom stock zerado manualmente + auditoria em `stock_adjustments`
