# R-177: Test Coverage Integrity

**Status: Ativa**
**Contexto: Automação e CI**

## Definição
Sempre que uma nova estrutura de diretórios for criada (ex: `server/`, `api/`, `scripts/`) que contenha lógica de negócio ou utilitários, o agente **DEVE** validar e, se necessário, atualizar os arquivos de configuração de testes (`vitest.config.js`, `jest.config.js`, etc.) para incluir esses novos caminhos.

## Por que isso é importante?
- **Falsa Segurança:** Testes que existem no sistema de arquivos mas não rodam no CI criam uma ilusão de cobertura.
- **Regressões Silenciosas:** Mudanças podem quebrar lógicas "testadas" sem que o desenvolvedor perceba, pois o comando de teste global passa com sucesso (ignorando os arquivos fora do scan).

## Como Aplicar
1. Ao criar um novo diretório principal, cheque a chave `include` ou `testMatch` no arquivo de configuração do test runner.
2. Adicione o glob pattern correspondente (ex: `'server/**/*.test.js'`).
3. Execute `npm run test` e confirme no log de execução que a quantidade de arquivos de teste aumentou conforme o esperado.
