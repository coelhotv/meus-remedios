# R-165: Hermes URL polyfills — substituição incondicional + LiveURLSearchParams

## Contexto

Expo Go SDK 53 / Hermes implementa `URL` e `URLSearchParams` parcialmente. Duas armadilhas distintas afectam o Supabase JS v2 em React Native:

1. **URLSearchParams mutations** (`set`, `append`, `delete`) lançam `"not implemented"` em uso real mas não em probes sintéticos
2. **`url.searchParams` getter** cria objectos desvinculados — mutações não propagam para o URL

## Regras

### 1 — Substituição incondicional de URLSearchParams

```js
// ❌ Nunca — detection probe falha silenciosamente no Hermes
;(function() {
  if (typeof URLSearchParams !== 'undefined') {
    try { new URLSearchParams().set('_test','_val'); return } catch(e) {}
  }
  global.URLSearchParams = MyImpl
})()

// ✅ Sempre — substituição incondicional
;(function patchURLSearchParams() {
  function HermesURLSearchParams(init) { ... }
  HermesURLSearchParams.prototype.set = function(name, value) { ... }
  // ... todos os métodos em ES5 puro (sem class, sem Symbol.iterator)
  global.URLSearchParams = HermesURLSearchParams
})()
```

### 2 — LiveURLSearchParams para url.searchParams

O `postgrest-js` usa `url.searchParams.set/append` para TODOS os query params (select, filtros, order). O getter simples cria objectos desvinculados → mutações perdidas → PGRST125.

**Solução: LiveURLSearchParams com `_sync()`**

```js
;(function patchLiveURLSearchParams() {
  // splitHref: isola base+search+hash sem depender de getters patchados
  function splitHref(href) {
    var qIdx = href.indexOf('?'), hIdx = href.indexOf('#')
    // ... lógica de split
  }

  function LiveURLSearchParams(urlObj) {
    this._url = urlObj
    // Inicializar pares do href actual
    var qs = splitHref(urlObj.href).search.slice(1)
    this._pairs = parsePairs(qs)
  }

  LiveURLSearchParams.prototype._sync = function () {
    var newSearch = buildSearch(this._pairs) // encodeURIComponent de cada par
    var split = splitHref(this._url.href)
    this._url.href = split.base + newSearch + split.hash // href setter funciona no Hermes
  }

  LiveURLSearchParams.prototype.set = function (name, value) {
    // ... actualizar _pairs
    this._sync()
  }
  LiveURLSearchParams.prototype.append = function (name, value) {
    this._pairs.push([String(name), String(value)])
    this._sync()
  }
  // delete, get, getAll, has, toString, forEach ...

  Object.defineProperty(URL.prototype, 'searchParams', {
    get: function () { return new LiveURLSearchParams(this) },
    configurable: true,
    enumerable: false,
  })
})()
```

**Porquê funciona:**
- Cada `url.searchParams.set('select', ...)` cria um LiveURLSearchParams, muta `_pairs`, chama `_sync()` → actualiza `url.href`
- O próximo `url.searchParams.append('user_id', ...)` cria outro LiveURLSearchParams inicializado com o href já actualizado — os pares anteriores estão preservados
- `url.toString()` = `url.href` → URL completa com todos os params → PostgREST recebe query correcta

## Ordem de aplicação em polyfills.js

```
1. patchHermesURL()          — URL.prototype getters/setters
2. patchURLSearchParams()    — global.URLSearchParams = HermesURLSearchParams
3. patchLiveURLSearchParams() — URL.prototype.searchParams = LiveURLSearchParams
```

O passo 3 deve ser último porque sobrepõe o getter `searchParams` definido (ou não) pelo passo 1.
