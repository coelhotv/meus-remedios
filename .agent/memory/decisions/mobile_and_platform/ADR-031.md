# ADR-031 — Usar user_settings em Jobs Serverless

## Status
**ACCEPTED**

## Contexto

Durante o Sprint 6.5, o dispatcher de notificações falhava silenciosamente ao tentar resolver usuários via `auth.users`. A causa raiz: jobs serverless não têm acesso estável ao schema `auth` do Supabase no contexto de `service_role` padrão.

A tabela `user_settings` no schema `public` replica as informações necessárias (user_id, preferências, configurações) e é acessível via service_role sem restrições.

## Decisão

Todo código que roda em contexto serverless (Vercel functions, api/*.js, jobs agendados) DEVE:

1. Buscar dados de usuário via `user_settings` (schema `public`)
2. **NUNCA** usar `auth.users` em workers/jobs serverless
3. Se precisar de email ou metadados de auth: criar uma view ou sincronizar via trigger para `user_settings`

## Consequências

- Dispatcher de notificações corrigido — resolve usuários corretamente
- `user_settings` torna-se a fonte canônica de dados de usuário no contexto serverless
- Qualquer nova funcionalidade serverless que precise de dados de usuário deve usar `user_settings`
