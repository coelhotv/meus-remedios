# [ADR-044] Distribuição de modelos cavecrew (Opus / Sonnet / Haiku) + gate de confiança

## Contexto
Plano "Claude Pro" tem limites semanais de tokens. Fase 0 (3 sprints) consumiu sob padrão "1 opus arquiteto + 100% sonnet cavecrew" — sem retrabalhos em P.2/P.3, mas custo alto. Spec original previa haiku para ⭐ — não foi usado na prática.

Fase 1 precisa reduzir consumo sem perder qualidade.

## Decisão
Tripartir distribuição:

| Modelo | Quando usar |
|--------|-------------|
| 👤 **Opus (arquiteto)** | Decisões de adaptação, navigation cross-screen, integrações complexas (form state + bottom sheet + auto-fill), lógica cross-domain (delete deps), factory pattern, migração web (risco regressão), gates de validação. **Sempre revisa output dos cavecrew.** |
| 🤖 **Sonnet (cavecrew ⭐⭐)** | Componentes/hooks com lógica de estado, integration test pattern P.3, testes do factory — onde sutilezas de R-010/R-020/lint refs importam. |
| 🐦 **Haiku (cavecrew ⭐)** | Tasks **estritamente mecânicas**: componente puro sem state (Empty, Banner), template copy-paste em demo screen, delete + grep, export 1-linha. |

## Gate de confiança Haiku (Fase 1)
Primeiro spawn Haiku do projeto = M1.10 (seção em `MedicineDemoScreen`).
- **0 retrabalho** → expande p/ todas tasks ⭐ marcadas (M1.6, M2.3, M2.10, M3.7, M3.10).
- **1+ retrabalho** → recua tudo para Sonnet, ajusta spec, registra lição.

Lint round-trip (`rtk lint <file>`) obrigatório pós-spawn. Qualquer fix manual = retrabalho contabilizado.

## Critério "estritamente mecânico" (Haiku)
- Tem referência canônica direta no brief (ex: "copie estrutura do `Toast.jsx` linhas 80-90 trocando X por Y").
- Sem decisão de design.
- Sem state novo.
- Sem integração nova.

Se a task exige qualquer das acima → é Sonnet.

## Brief padrão (compartilhado entre Sonnet e Haiku)
Ver [R-230]. Os 6 itens são obrigatórios para ambos os modelos — Haiku recebe brief mais curto, mas com a mesma estrutura.

## Consequências esperadas
- Economia de tokens em ~30% nas tasks ⭐ comparado a usar Sonnet 100%.
- Risco controlado pelo gate de confiança M1.10.
- Padrão reusável em Fases 2 (Protocolos) e 3 (Estoque) se gate Fase 1 confirmar viabilidade.
