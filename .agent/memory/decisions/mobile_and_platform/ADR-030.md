## Context

A migração de `checkReminders`, `checkStockAlerts` e `runDailyDigest` para o dispatcher é a mudança mais arriscada da Fase 6. Se o dispatcher falhar, lembretes param para **todos os usuários em produção**.

## Decision

Cada job migrado verifica:

```js
const USE_DISPATCHER = process.env.USE_NOTIFICATION_DISPATCHER !== 'false'
```

**Default:** `true` (dispatcher ativo)  
**Fallback:** Telegram direto via adapter legado  
**Remoção do fallback:** somente após 2 semanas de operação estável  

Se houver problema em produção:
1. Maintainer seta `USE_NOTIFICATION_DISPATCHER=false` no Vercel (sem deploy)
2. Próxima execução de cron usa Telegram legado
3. Zero downtime — reminders continuam chegando

## Alternatives Rejected

1. **Sem rollback flag** — risco inaceitável
   - Deploy falho = lembretes param para todos
   - Revert exige novo deploy (minutos de downtime)
   - Sem caminho rápido de mitigation

2. **Rollback via database flag** — mais lento
   - Requer lógica no job para ler DB
   - Latência extra em cada execução
   - Env var é mais rápido e simples

## Consequences

✅ Rollback **imediato** sem deploy (apenas env var no Vercel)  
✅ Telegram legado permanece como fallback (2 semanas mínimas)  
✅ Código defensivo: todos os jobs gateados pela flag  

⚠️ Custo: ~10 linhas extras por job migrado  
⚠️ Operacional: requer maintainer aware do rollback procedure  

## Linked Rules

- **R-041** (nunca process.exit())
- **R-086** (sempre res.status(code).json())
- **R-087** (logging estruturado)

## Implementation Notes

- Sprint 6.1 prepara banco (sem flag necessária)
- Sprint 6.2 prepara dispatcher (sem flag)
- **Sprint 6.4** adiciona a flag em cada job migrado
- Deploy com `USE_NOTIFICATION_DISPATCHER=true` (default)
- Fallback legado removido após 2 semanas de estabilidade (Sprint 6.6 ou posterior)

## Removal Criteria

Flag removida somente quando:
- [ ] 14+ dias de operação estável com dispatcher ativo
- [ ] Nenhum incident relacionado a push fallback
- [ ] Todos os logs de dispatch visíveis e sadios
- [ ] Maintainer aprova remoção do fallback
- [ ] Novo PR removendo código legado + flag

## Acceptance Criteria

- [x] Aprovado pelo maintainer antes de Sprint 6.4 merge
- [ ] Feature flag presente em `checkReminders`, `checkStockAlerts`, `runDailyDigest`
- [ ] Fallback para Telegram via adapter legado funciona
- [ ] Logs distinguem entre dispatcher path vs legacy path
- [ ] `USE_NOTIFICATION_DISPATCHER=false` em staging testado com sucesso
