# ADR-001: Gradual Redesign Rollout via Feature Flags

**Status:** Accepted
**Date:** 2026-03-24
**Source:** PR #416 — feat(redesign): infraestrutura de rollout gradual
**Tags:** architecture, feature-flags, rollout, redesign

## Context

The dosiq project requires a complete visual overhaul (Santuário Terapêutico design) to improve UX and modernize the interface. However, rolling out this change to all users immediately would introduce visual disruption and complicate validation during design research sessions.

The team needs to develop Waves 0-12 of the new design system in parallel with the current neon/cyberpunk interface, allowing:
- Private design research sessions with selective users
- Zero visual impact on regular users until design is validated
- Incremental rollout strategy

## Options Considered

1. **Branch-based parallel development** — maintain separate branches for old/new UI
   - ❌ Difficult to merge, high code duplication, merge conflicts

2. **Environment-based toggle (staging vs production)** — separate deployments
   - ❌ Requires multiple production deployments, operational overhead

3. **Feature flag with localStorage + URL parameter** (chosen)
   - ✅ Single codebase, single deployment
   - ✅ Granular per-user control
   - ✅ CSS scoped, zero visual leakage
   - ✅ URL param for research sessions, localStorage for persistence

## Decision

Implement a **feature flag infrastructure** with the following characteristics:

### Activation Mechanism
- **URL parameter:** `?redesign=1` toggles redesign mode
- **Persistence:** Sets `localStorage.mr_redesign_preview = '1'`
- **Deactivation:** `?redesign=0` or clear localStorage
- **CSS Scope:** All new design tokens live under `[data-redesign="true"]` selector on `.app-container`

### Implementation Details

| Component | Purpose | File |
|-----------|---------|------|
| `RedesignContext.jsx` | State management + flag logic | `src/shared/contexts/` |
| `useRedesign()` hook | Provide `{ isRedesignEnabled, toggleRedesign }` | `src/shared/hooks/` |
| `App.jsx` wrapper | Apply `data-redesign` attribute based on flag | `src/App.jsx` |
| `tokens.redesign.css` | W0 + W1 design tokens scoped | `src/shared/styles/` |

### CSS Scoping Strategy

```css
/* tokens.redesign.css */
[data-redesign="true"] {
  --color-primary: #006a5e;      /* health green */
  --color-secondary: #005db6;    /* clinical blue */
  --color-surface: #f8fafb;      /* light background */
  --shadow-sm: 0 1px 2px rgba(...); /* ambient shadows */
  /* ... more tokens ... */
}

/* Backward compat aliases for non-redesign mode */
[data-redesign="false"],
:not([data-redesign]) {
  --color-primary: #00ff00;      /* original neon */
  --color-secondary: #ff00ff;    /* original cyberpunk */
  /* ... original tokens ... */
}
```

## Consequences

### Positive
- ✅ Single codebase, single deployment pipeline
- ✅ Zero visual impact on regular users until redesign enabled
- ✅ Research team can share URL with selective users for testing
- ✅ Easy rollback (just disable flag)
- ✅ Incremental wave development without blocking production

### Negative
- ❌ CSS file size increases (both old + new styles shipped)
- ⚠️ Must maintain two design systems in parallel during transition
- ⚠️ CSS specificity must be careful to avoid unintended scope leakage

## Rollback Plan

If the redesign validation fails:
1. Set `localStorage.mr_redesign_preview = '0'` globally
2. Remove `[data-redesign="true"]` from app root
3. No code changes needed

If specific waves fail validation:
1. Disable specific wave CSS imports in `tokens.redesign.css`
2. Revert relevant PR
3. Feature flag remains for other waves

## Testing

- Verify app without `?redesign=1` → 100% neon/cyberpunk appearance unchanged
- Verify with `?redesign=1` → colors/typography change to new system
- Verify URL param persists in localStorage
- Verify CSS computed values differ with/without flag via DevTools
- Verify toggle in Settings (dev-mode) works and persists
