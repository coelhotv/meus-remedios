# [ADR-045] Factory location para repositórios CRUD compartilhados — `@dosiq/core/repositories/`

## Status
Accepted (2026-05-16)

## Contexto

Fase 1 (R-231) estabeleceu o padrão **factory + DI** para CRUD compartilhado web↔mobile, e instanciou a primeira fábrica (`createMedicineRepository`) em `packages/core/src/repositories/`.

A spec original da Fase 2 (`EXEC_SPEC_FASE2_PROTOCOLOS.md` linhas 87-89, T3.1/T3.2) propôs um caminho **divergente** para os factories de Protocolos e Treatment Plans:

```
packages/shared-data/src/services/createProtocolRepository.js
packages/shared-data/src/services/createTreatmentPlanRepository.js
```

Dois locations diferentes para a **mesma classe de código** quebra a discoverability e cria fricção para os próximos CRUDs (Fase 3 Estoque, etc.).

## Decisão

**Todos os repositórios CRUD compartilhados ficam em `@dosiq/core/repositories/`.**

Layout canônico:

```
packages/core/src/
  repositories/
    index.js
    createMedicineRepository.js          ← Fase 1 G2
    createProtocolRepository.js          ← Fase 2 G2 (a criar)
    createTreatmentPlanRepository.js     ← Fase 2 G2 (a criar)
    createStockRepository.js             ← Fase 3 G2 (futuro)
    __tests__/
      createMedicineRepository.test.js
      createProtocolRepository.test.js
      createTreatmentPlanRepository.test.js
```

`packages/shared-data/` continua existindo para **dados estáticos** (ex: catálogos ANVISA, listas de UF, etc.) — não para serviços CRUD.

## Justificativa

1. **R-231 já estabelecida**: Mover seria duplicar padrão sem ganho.
2. **`@dosiq/core` já é importado por web + mobile + api + server/bot**: zero overhead de novo workspace.
3. **Validação Zod canônica já vive em `@dosiq/core/schemas/`**: factory + schema juntos = colocation lógica.
4. **Parity tests rodam via vitest do web** (configurado para varrer `packages/core/**/*.test.js`): infra já pronta, não precisa replicar em `shared-data`.
5. **`shared-data` tem semântica diferente**: dados estáticos vs serviços com I/O e validação. Misturar dilui o propósito do pacote.
6. **Discoverability**: dev novo procura "createXRepository" em 1 lugar.

## Alternativas consideradas

| Opção | Por que descartada |
|-------|--------------------|
| `packages/shared-data/src/services/` | Spec original. Divergente do precedente Fase 1; mistura tipos de conteúdo no pacote. |
| `packages/repositories/` (novo workspace) | Overhead de novo workspace sem benefício prático; `@dosiq/core` já cumpre o papel. |
| Manter ad-hoc por app | Anti-pattern explícito do R-231 (gera drift web↔mobile). |

## Consequências

**Positivas:**
- 1 padrão de location para todos os CRUDs futuros.
- Reuso imediato da infra de testes (vitest pega `packages/core/**/*.test.js`).
- Fase 2 começa com decisão arquitetural já feita (sem debate no meio do gate).

**Negativas:**
- `EXEC_SPEC_FASE2_PROTOCOLOS.md` precisa ser corrigido (T3.1, T3.2, T3.3, T3.4 referenciam `shared-data`).
- Caso `@dosiq/core` cresça demais no futuro, faz sentido splittar — mas só com pelo menos 5+ factories convivendo (não antecipar).

## Aplicação

- Fase 2 T3.1: `createProtocolRepository` em `packages/core/src/repositories/createProtocolRepository.js`
- Fase 2 T3.2: `createTreatmentPlanRepository` em `packages/core/src/repositories/createTreatmentPlanRepository.js`
- Fase 3 (Estoque): seguir mesmo padrão.
