# [R-230] Brief padrão para spawn cavecrew (sub-agentes)

## Definição
Todo spawn de cavecrew (`caveman:cavecrew-builder`, sonnet ⭐⭐ ou haiku ⭐) DEVE receber um brief contendo os 6 itens abaixo. Faltando qualquer item, o risco de retrabalho cresce ~3x (medido empiricamente em Fase 0).

## Os 6 itens obrigatórios
1. **Read-only refs absolutas** para todos arquivos a consultar — incluir paths completos (não relativos), incluindo mocks PNG quando aplicável e doc canônica (`docs/reference/FORM_KIT.md` ou similar).
2. **Path absoluto** do arquivo a criar/modificar.
3. **Contrato exato** — props/API esperados, com exemplos de chamada.
4. **Regras críticas explícitas** — R-010 (ordem hooks), R-020 (zero `new Date()` direto), R-022 (dose max 100), lint `react-hooks/refs`, `react-hooks/set-state-in-effect`, zero color literals, `StyleSheet.create()`.
5. **Output esperado** — código + ponto único de integração.
6. **Sem commits** — apenas Write/Edit.

## Pós-spawn obrigatório (arquiteto)
- `rtk lint <file>` — qualquer erro = retrabalho contabilizado.
- `rtk jest <pattern>` quando aplicável.
- Fix manual de qualquer issue residual antes do commit.

## Histórico
- Fase 0 P.1: 2 retrabalhos cavecrew sonnet (brief inicial omitiu 2 dos 6 itens).
- Fase 0 P.2: 0 retrabalhos (brief completo).
- Fase 0 P.3: 0 retrabalhos (brief completo).
- Padrão validado e adotado como obrigatório.

## Diferença Sonnet vs Haiku
- **Sonnet (⭐⭐)**: tarefas com lógica de state, sutilezas de hooks, decisões de design não óbvias.
- **Haiku (⭐)**: tarefas **estritamente mecânicas** com referência canônica direta (ex: "copie estrutura do `Toast.jsx` linhas 80-90 trocando X por Y"). Brief mais curto, mas mantém os 6 itens. Se exige decisão de design não óbvia, é Sonnet.
