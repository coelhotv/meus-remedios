# R-237 — Wave orchestration com dependency graph antes de spawn cavecrew

**Regra**: ao planejar sprint com 5+ tasks que admitem paralelismo, mapear dependency graph ANTES de spawn:

1. Listar todas as tasks
2. Para cada task, identificar inputs (tasks que precisam estar done antes)
3. Agrupar tasks sem deps mútuas em "waves" paralelas
4. Spawn por wave; aguardar wave completar antes da próxima
5. Reservar Opus inline pra tasks sensíveis (integração, lógica complexa, decisões arquiteturais)
6. Sonnet pra tasks mecânicas complexas (espelhar pattern, refactor escopo claro)
7. Haiku pra tasks mecânicas simples (tests, find/replace, docs cross-ref)

**Why**: Fase 2.5 entregou 12 tasks em 4 waves com 9 spawns paralelos. Sem orchestration, seriam 12 spawns sequenciais ou Opus inline em tudo. Economia estimada: 60% tokens Opus, 40% tempo total.

**How to apply**:

```
Wave 1 (inline + spawn paralelo): tasks sem deps
Wave 2 (após Wave 1): tasks que dependem de outputs Wave 1
Wave 3: idem após Wave 2
...
Wave Final (inline Opus): integração que consome outputs de múltiplas waves
```

Brief de cada spawn (R-230) DEVE:
- Listar arquivos read-only refs absolutos
- Path absoluto do arquivo a criar/modificar
- Contrato exato (props, return shape)
- Regras críticas R-NNN aplicáveis
- VALIDAÇÃO pós-code com comandos exatos
- "NÃO COMMITAR" explícito (Opus consolida + commita no fim da wave)

**Não aplicar quando**:
- Sprint tem <5 tasks (orchestration overhead > ganho)
- Tasks têm deps lineares 1→2→3→4 (waves degeneram em sequencial)
- Sprint exploratório (especificação evolui durante execução; deps não claras upfront)

**Exemplo concreto** (Fase 2.5 — 12 tasks, 4 waves):
- Wave 1 inline (Opus): T1 helper + barrel
- Wave 2 spawn paralelo (6 agents): T2 Haiku, T3 Sonnet, T4 Sonnet, T7 Sonnet, T8 Sonnet, T11 Haiku
- Wave 3 spawn (1): T5 Sonnet (depende T4)
- Wave 4 spawn (1): T6 Sonnet (depende T5)
- Wave 5 inline (Opus): T9 + T10 (integração que consome T6, T7, T11)
- T12 spawn Haiku (paralelo a Wave 5; doc-only, sem deps)
