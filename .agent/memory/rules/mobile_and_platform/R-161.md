# R-161 — Mobile Expo development: iCloud repos need file system workarounds

**Trigger count:** 1 (2026-04-12)  
**Related AP:** AP-H07 (iCloud Watchman permission error)  
**Wave:** H4 (Mobile Scaffold, Phase 4)  
**Pack:** adherence-reporting-mobile  

---

## Rule

When developing Expo mobile apps in a repository stored in **iCloud Cloud Documents** (macOS path: `~/Library/Mobile Documents/com~apple~CloudDocs/git/...`), `npx expo start` fails with Watchman permission errors.

**Solutions:**
1. Clone repo locally (outside iCloud) — recommended for maintainers with Apple/Google accounts
2. Create `.watchmanconfig` with `{}` in repo root — partial fallback (may not fully resolve)
3. Simulator validation is **maintainer-gated** — agents cannot perform simulator validation without account access

---

## Why

**iCloud Cloud Documents** applies file system restrictions to keep files synced with Apple's servers. This prevents:
- Watchman (Expo's Metro bundler file watcher) from monitoring `node_modules/` changes
- `operation not permitted` errors when attempting to access `node_modules/expo` or similar subdirectories

**Expo constraint:**
- Older Expo versions had `--no-watchman` flag to disable Watchman — no longer supported in current versions
- No built-in fallback or environment variable to disable Watchman entirely

---

## How to Apply

### For Maintainers (With Apple/Google Accounts) — Running `npx expo start` Locally

**Option 1: Clone repo outside iCloud (Recommended)**
```bash
# Clone repo to local filesystem (outside iCloud sync)
git clone https://github.com/coelhotv/dosiq.git ~/local/dosiq
cd ~/local/dosiq/apps/mobile

# npx expo start now works (Watchman has full file system access)
npx expo start
# ✓ Metro bundler starts, Watchman monitoring active
```

**Option 2: Create .watchmanconfig (Partial Fallback)**
```bash
# In repo root where iCloud repo lives
echo "{}" > .watchmanconfig

# Try again
npx expo start
# May still fail due to iCloud path restrictions (not guaranteed)
```

### For Agents (No Account Access) — Simulator Validation Blocked

**Status:** Simulator validation is **maintainer-gated**.

From H4 acceptance criteria:
```
"Bloqueio humano: contas Expo/Apple/Google + EAS secrets pendentes.
O agente pode implementar tudo menos EAS build e validacao em simuladores."
```

**Agent responsibility:**
- ✅ Implement scaffold (code complete)
- ✅ Ensure local web builds work (`npm run build` passes)
- ✅ Ensure tests pass (`npm run test:critical` passes)
- ❌ Simulator validation (requires Apple Developer account + local clone or Expo EAS)
- ❌ EAS build execution (requires Apple/Google account + secret configuration)

---

## Prevention Checklist

When working with Expo mobile development in future waves:

1. **Know the constraint:** iCloud repos cannot run `npx expo start` reliably
2. **Early detection:** If you see `Watchman error: operation not permitted`, check the working directory path
3. **Path check:** `pwd` — if output contains `Library/Mobile Documents`, this is the iCloud constraint
4. **Workaround expectation:** Only `.watchmanconfig` is tested; might not fully resolve iCloud restrictions
5. **Gate identification:** If you're an agent without Apple/Google account access:
   - Don't spend time troubleshooting Watchman errors
   - This is a documented maintainer-gated validation step (R-161)
   - Focus on code-level validation (tests, lint, build)
6. **For maintainers validating simulator:**
   - Clone repo locally (outside iCloud) before running `npx expo start`
   - Test on iOS Simulator or Android Emulator from local clone
   - Report validation results back to the agent
