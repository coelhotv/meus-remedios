# R-168: Custom Polyfill (Estratégia A) é a Solução Definitiva para Hermes

**Criado:** 2026-04-14 | **Origem:** Spike whatwg-url-without-unicode confirmou que alternativas não funcionam

---

## Contexto

O Hermes engine (Expo SDK 53) implementa `URL` e `URLSearchParams` parcialmente.
O Supabase `postgrest-js` depende 100% destas APIs para construir queries REST.

## Alternativas testadas e rejeitadas

| Alternativa | Crash | Causa raiz |
|-------------|-------|-----------|
| `react-native-url-polyfill` v2/v3 | `Cannot read property 'get' of undefined` | Wrapper acessa `NativeModules.BlobModule.getConstants()` — inexistente no Expo Go |
| `whatwg-url-without-unicode` directo | `Cannot read property 'get' of undefined` | Usa `Symbol.for("[webidl2js] constructor registry")` — nunca inicializado no Hermes |
| `whatwg-url` (versão completa) | Mesmo + deps mais pesadas | Mesmo problema WebIDL |

## Por que o polyfill custom funciona

1. **Não substitui a classe URL** — apenas adiciona getters/setters ao `prototype`
2. **`needsPatch()` defensivo** — testa cada getter antes de substituir
3. **`parseHref()` baseado em regex** — extrai campos a partir de `URL.href` (que funciona no Hermes)
4. **`_searchPairs` backing store** — evita dependência de `URL.search` setter
5. **`toString()` override** — reconstrói URL a partir de base + params, remove trailing slash

## Implicações

- O polyfill **DEVE ter testes automatizados** (recomendação B1 — ALTA prioridade)
- Qualquer update de Expo/Hermes deve ser testado contra o polyfill
- O ficheiro `polyfills.js` deve ser importado PRIMEIRO em `index.js` (R-162)
- Pin da versão Supabase (`2.91.0`) recomendado até testes existirem

## Spike preservado

Branch `spike/whatwg-url-polyfill` no GitHub para referência futura.
