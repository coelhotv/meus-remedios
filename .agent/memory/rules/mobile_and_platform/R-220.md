# R-220 — 3-Layer Notification Architecture (L1, L2, L3)

## Regra

O sistema de notificações do Dosiq DEVE seguir a estrutura de 3 camadas para garantir que novos canais e formatos sejam adicionados sem refatoração da lógica de negócio.

### Camada 1: Lógica de Negócio (L1)
*   **Responsabilidade**: Decidir QUANDO notificar e fornecer DADOS CRUS.
*   **Localização**: `server/bot/tasks.js`, `api/notify.js`.
*   **Restrição**: NUNCA deve formatar strings, incluir emojis, escapar Markdown ou decidir ações de interação (UI).
*   **Exemplo**: Passar `{ medicineName: 'Paracetamol', quantity: 1 }` em vez de `'Tome 1 pilula de *Paracetamol* 💊'`.

### Camada 2: Apresentação / Builder (L2)
*   **Responsabilidade**: Transformar dados crus em um `NotificationPayload` canônico.
*   **Localização**: `server/notifications/payloads/`.
*   **Propriedades**: Títulos, corpos (MarkdownV2), emojis, saudações, e `actions[]` semânticas.
*   **Restrição**: NÃO deve conhecer canais específicos (Telegram vs Push). Deve entregar um contrato abstrato.
*   **Saída**: Objeto validado pelo `notificationPayloadSchema`.

### Camada 3: Entrega / Delivery (L3)
*   **Responsabilidade**: Despachar o payload para os canais do usuário.
*   **Localização**: `server/notifications/channels/`, `dispatcher/`.
*   **Ações**: Resolver canais, aplicar supressão (R-200), mapear `actions[]` para botões nativos (Telegram inline keyboards), truncar `callback_data` (R-030).
*   **Restrição**: NÃO deve modificar o conteúdo da mensagem (Markdown deve chegar pronto de L2).

## Por que?
Evita o "vazamento" de lógica de UI (Telegram/Push) para o core do sistema. Permite que a mesma notificação seja visualizada no Inbox Web (Markdown), Telegram e Push Notifications com paridade visual e funcional.
