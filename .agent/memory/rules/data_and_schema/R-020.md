# R-020 — Timezone — Local Dates & SP Hardening [CRITICAL]

## Rule
1. **ALWAYS** use `parseLocalDate(dateStr)` from `@utils/dateUtils` or `new Date(dateStr + 'T00:00:00')`.
2. **ALWAYS** use the "Canadian Hardening" pattern in `getSaoPauloTime` and `getStartOfDayISO` to ensure a stable `YYYY-MM-DD` string for parsing.
3. **NEVER** use `new Date('YYYY-MM-DD')` — this creates UTC midnight, which is 21:00 of the PREVIOUS day in GMT-3.

## SP Time Hardening (The Canadian Trick)
To force São Paulo time across different device locales/engines, use `en-CA` which outputs an unambiguous `YYYY-MM-DD` format:

```javascript
// STABLE & SAFE (Canadian Pattern)
export function getSaoPauloTime(date = new Date()) {
  return new Date(
    date.toLocaleString('en-CA', {
      timeZone: 'America/Sao_Paulo',
      hour12: false,
    }).replace(', ', 'T') // Force ISO-like format
  )
}
```

## Why
- **Local Date Bug**: `new Date('2026-02-18')` creates 2026-02-17T21:00:00 in GMT-3. Adherence scores were attributed to the wrong day.
- **Parser Ambiguity**: `en-US` locale can output `M/D/YYYY` or `MM/DD/YYYY` with AM/PM, which is fragile. `en-CA` is ISO-consistent and eliminates AM/PM logic.
- **Node 24 Regression (2026-05-02)**: In Node 24, `en-CA` locale added `p.m.` instead of `PM`, causing `new Date()` to return `Invalid Date`. Hardening with `hour12: false` and replacing `, ` with `T` is now MANDATORY.

## Examples
```javascript
// Correct parsing of string
import { parseLocalDate } from '@utils/dateUtils'
const date = parseLocalDate('2026-02-18')

// Correct SP time acquisition
const nowInSP = getSaoPauloTime()

// WRONG — midnight UTC = 21:00 Feb 17 in GMT-3
const date = new Date('2026-02-18')
```

## Applies to
Any date string in YYYY-MM-DD format from Supabase, API responses, or user input. Also applies to any internal date shifting logic.
