# R-159 — Monorepo workspace changes: always commit updated lock file

**Pack:** infra-api  
**Introduced:** Wave H4 (Mobile Scaffold, 2026-04-12)  
**Incidents Prevented:** 1 (GitHub Actions npm ci sync failure)  
**Related Anti-Pattern:** AP-H05  

---

## Rule

When modifying `package.json` in a monorepo with npm workspaces (adding new workspace, updating dependencies), **always commit the updated `package-lock.json`** alongside your package.json changes.

### Why

**npm ci (CI Install) requires synchronization:**
- Local development: `npm install` automatically updates lock file
- CI/CD (GitHub Actions): `npm ci` is strict — it requires package.json and package-lock.json to be **perfectly in sync**
- Consequence: if you commit only package.json, CI fails with `EUSAGE` error and blocks all downstream jobs

### Pattern

```bash
# Workflow for adding new workspace or changing dependencies

1. Modify package.json (add new workspace, change versions, etc.)
2. Run: npm install              # Updates package-lock.json
3. Validate: npm run validate    # Ensure no lint/test regressions
4. Stage BOTH:
   git add package.json package-lock.json
5. Commit: git commit -m "feat: add workspace" (or fix/chore)
6. Push: git push
7. Verify: GitHub Actions passes on first run
```

## How

### Step 1: Identify Workspace/Dependency Changes

```bash
# Any of these trigger R-159:
# - Adding new workspace (apps/mobile/)
# - Changing root dependency versions
# - Updating workspace package.json
```

### Step 2: Install & Verify

```bash
# Update lock file
npm install

# Validate locally BEFORE committing
npm run lint
npm run build
npm run test:critical
```

### Step 3: Commit Both Files

```bash
# Check what changed
git status
# Expected output:
#   modified:   package.json
#   modified:   package-lock.json

# Stage BOTH
git add package.json package-lock.json

# Commit with semantic message
git commit -m "feat(workspace): add mobile app"
# or
git commit -m "chore: update package-lock.json for mobile workspace"
```

### Step 4: Verify CI Passes

```bash
# Push
git push

# Monitor GitHub Actions
gh pr checks <PR_NUMBER>

# Expected: Lint → Smoke → Build → Critical (all pass)
# NOT: skipped jobs because npm ci failed
```

## Anti-Pattern (Violation)

```bash
# ❌ WRONG — forget to commit lock file
git add apps/mobile/package.json
npm install  # ← lock file updated
git commit -m "feat: add mobile app"
git push
# Result: GitHub Actions EUSAGE error, CI blocked

# ✓ CORRECT — include lock file
git add apps/mobile/package.json package-lock.json
npm install  # ← lock file updated
git commit -m "feat: add mobile app"
git push
# Result: GitHub Actions passes, CI chain succeeds
```

## FAQ

**Q: Why not use `.gitignore` for lock files?**  
A: Lock files MUST be committed in npm workspaces. They ensure exact reproducibility in CI/CD. Ignoring them breaks `npm ci`.

**Q: What if I only changed one dependency version?**  
A: Same rule applies. Any `package.json` change requires lock file commit. Run `npm install`, then commit both.

**Q: Can I use `npm ci --legacy-peer-deps`?**  
A: Never in CI. That hides peer dep conflicts. Resolve them properly locally, then commit lock file.

**Q: What if lock file has merge conflicts?**  
A: Resolve conflicts locally, run `npm install` to regenerate lock file, then commit the clean version.
