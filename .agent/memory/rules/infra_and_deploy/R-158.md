# R-158 — Monorepo workspace packages: use dependency ranges, not exact versions

**Pack:** infra-api  
**Priority:** High  
**Introduced:** Wave H4 (Mobile Scaffold, 2026-04-12)  
**Incidents Prevented:** 1 (ERESOLVE conflict on `apps/mobile`)  
**Related Anti-Pattern:** AP-H04  

---

## Rule

When creating a new workspace package in a monorepo (npm workspaces), declare shared library versions using **semver ranges** (`^X.Y.Z`), not exact versions (`X.Y.Z`).

### Why

1. **Peer dependency resolution:** Dev tools (jest, testing libraries) declare their own peer deps on shared libraries
2. **npm ERESOLVE:** npm 7+ strictly validates that all peer deps can be satisfied by the same version
3. **Exact versions block:** If root has `react@19.0.0` and new package declares `react@19.0.0`, but a test tool requires `react@^19.2.5`, npm fails with `ERESOLVE unable to resolve dependency tree`
4. **Ranges solve it:** Using `react@^19.2.5` allows npm to pick a version that satisfies **both** root constraints and dev tool peer deps

---

## How

### Step 1: Check Root Versions

```bash
# Check what root project uses
grep -A 20 '"dependencies"' package.json

# Example (meus-remedios root):
# "react": "19.0.0" (or "^19.2.5")
# "typescript": "^5.0.0"
```

### Step 2: Check New Package's Dev Tool Peer Deps

```bash
# For each dev tool you'll add, check its peer deps
npm view jest-expo@53.0.0 peerDependencies
npm view @testing-library/react-native@13.2.0 peerDependencies

# Example:
# jest-expo@53.0.0 peer: react@">=18.0", react-native@0.79.x
# @testing-library/react-native@13.2.0 peer: react@">=18.0", react-test-renderer@>=18.0
```

### Step 3: Write Aligned Versions

**Pattern:**

```json
{
  "dependencies": {
    "react": "^19.2.5",                    // ← Use RANGE to satisfy peer deps
    "react-native": "0.79.2",              // ← Already satisfies jest-expo peer
    "expo": "^53.0.0",                     // ← Ranges OK for non-root dependencies
    "expo-secure-store": "^14.0.1"
  },
  "devDependencies": {
    "jest": "^29.7.0",                     // ← Range, test tools often update
    "jest-expo": "~53.0.0",                // ← Tilde for test tools (stability)
    "@testing-library/react-native": "^13.2.0"
  }
}
```

**Why the pattern:**
- Shared libs (react, react-native, etc.): use **^** (compatible with minor/patch updates)
- New workspace-only libs: use **^** or **~** freely (no peer dep conflicts at root)
- Dev tools: use **^** or **~** based on stability; prefer **~** for `jest-expo` because Expo often pins minor versions

### Step 4: Validate Before Commit

```bash
# Trigger ERESOLVE locally (DON'T use --legacy-peer-deps)
npm install

# If npm install succeeds, you're safe. If it fails:
# → Read the error, identify which peer deps conflict
# → Bump the conflicting library to a compatible range
# → npm ci (clean install) again

# Then validate:
npm run lint
npm run test:critical
npm run build
```

## Anti-Pattern (Violation)

```json
// ✗ WRONG — exact versions on shared libs
{
  "dependencies": {
    "react": "19.0.0",                   // Exact: blocks peer deps
    "react-native": "0.79.2",
    "expo": "53.0.0"                     // Exact for workspace-only lib (OK)
  },
  "devDependencies": {
    "@testing-library/react-native": "13.2.0"  // Exact (blocks if peer deps don't match)
  }
}

// ✓ CORRECT — ranges on shared libs
{
  "dependencies": {
    "react": "^19.2.5",                  // Range: satisfies peer deps
    "react-native": "^0.79.2",           // Range: compatible with jest-expo
    "expo": "^53.0.0"                    // OK to keep range for new workspaces
  },
  "devDependencies": {
    "@testing-library/react-native": "^13.2.0"  // Range for flexibility
  }
}
```

## Checklist (New Workspace Package)

- [ ] New subpackage directory created? (e.g., `apps/mobile/`)
- [ ] package.json draft written?
- [ ] BEFORE npm install: check dev tool peer deps (`npm view X peerDependencies`)
- [ ] Update all shared lib versions to ranges: `"react": "^X.Y.Z"`, not `"19.0.0"`
- [ ] Run `npm install` locally to validate ERESOLVE early
- [ ] If ERESOLVE fails: identify conflict, bump conflicting lib to compatible range, retry
- [ ] Lint, test:critical, build all pass?
- [ ] Commit, push
- [ ] Verify Vercel deployment succeeds (no ERESOLVE on CI)
