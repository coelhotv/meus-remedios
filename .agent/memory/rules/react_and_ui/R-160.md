# R-160 — React and React-DOM: pin to exact same version

**Pack:** react-hooks  
**Introduced:** Wave H4 (Mobile Scaffold, 2026-04-12)  
**Incidents Prevented:** 1 (version mismatch test failure)  
**Related Anti-Pattern:** AP-H06  

---

## Rule

When declaring `react` and `react-dom` in any workspace or root `package.json`, **pin both to the exact same version number** (no semver ranges like `^`, `~`, or `*`).

### Why

1. **Paired package:** React and React-DOM must be perfectly synchronized — they share internal state
2. **Patch version sensitivity:** Unlike most libraries, different patch versions (19.2.5 vs 19.2.3) have incompatible internals
3. **npm range drift:** Semver ranges allow npm to install different patch versions, breaking the pairing
4. **Test/runtime failure:** Mismatched versions cause:
   - Tests fail with "Incompatible React versions" error
   - CI/CD pipeline blocked until fixed
   - Runtime errors in production

### Pattern

```json
// ❌ WRONG — ranges allow version drift
{
  "dependencies": {
    "react": "^19.2.0",       // Can be 19.2.5
    "react-dom": "^19.2.0"    // Can be 19.2.3
  }
}

// ✓ CORRECT — exact versions, identical
{
  "dependencies": {
    "react": "19.2.5",
    "react-dom": "19.2.5"
  }
}
```

## How

### Step 1: Check Current Version

```bash
npm view react@latest version
# 19.2.5
```

### Step 2: Update package.json

**Both must be identical, no ranges:**

```json
{
  "dependencies": {
    "react": "19.2.5",        // ← Exact
    "react-dom": "19.2.5",    // ← Same, exact
    "other-lib": "^1.0.0"     // ← Ranges OK for other libs
  },
  "devDependencies": {
    "@types/react": "^19.2.5",  // Types can use ranges
    "@types/react-dom": "^19.2.3"
  }
}
```

### Step 3: Install & Verify

```bash
npm install

# Verify versions match
npm list react react-dom
# meus-remedios@4.0.0
# +-- react@19.2.5
# +-- react-dom@19.2.5    ← Must be identical
```

### Step 4: Run Tests

```bash
npm run test:smoke      # Must pass
npm run test:critical   # Must pass
```

### Step 5: Commit

```bash
git add package.json package-lock.json
git commit -m "fix: pin React and React-DOM to exact same version (19.2.5)"
```

## Anti-Pattern (Violation)

```bash
# ❌ WRONG — version mismatch after npm install
$ npm install
# npm resolves:
#   react@19.2.5
#   react-dom@19.2.3

$ npm run test:smoke
# Error: Incompatible React versions... react: 19.2.5, react-dom: 19.2.3

# ✓ FIX — pin both identically
{
  "react": "19.2.5",
  "react-dom": "19.2.5"
}
```

## Workspace Packages (Mobile)

Same rule applies to any workspace:

```json
// apps/mobile/package.json
{
  "dependencies": {
    "react": "19.2.5",        // ← Must match root
    "react-dom": "19.2.5",    // ← Must match root
    "react-native": "0.79.2"  // Different lib, fine with different version
  }
}
```

## FAQ

**Q: Why not use ranges like most other libraries?**  
A: React/React-DOM are tightly coupled internal APIs. Patch versions differ at the bytecode level, not just semantically.

**Q: What if I want to upgrade React?**  
A: Update both to the new version (exact), test locally, commit, push.

**Q: Can I use `^19` for flexibility?**  
A: No. Never. React/React-DOM MUST be identical. If you need flexibility, upgrade both together.

**Q: What about `@types/react` and `@types/react-dom`?**  
A: Types can use ranges (`^19.2.5`), they don't affect runtime. But keep actual react/react-dom exact.
